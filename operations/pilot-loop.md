# Foundry Pilot Loop

## Purpose

The Pilot Loop defines the first minimally complete operational embodiment of The Foundry operating beneath Citadel governance.

The objective is not theoretical orchestration.

The objective is executable institutional flow.

---

# Core Principle

The ecosystem must pressure-test doctrine against operational reality.

Institutional theory without embodiment eventually drifts into abstraction.

---

# Pilot Objective

Execute a complete operational cycle capable of:

- receiving intake
- classifying consequence
- staffing professions
- activating execution
- producing artifacts
- performing verification
- escalating uncertainty
- assigning final disposition
- preserving continuity

---

# Initial Scope Constraints

The Pilot Loop should remain:

- small
- observable
- reconstructable
- governance-aware
- artifact-centric

Avoid:

- premature autonomy
- excessive staffing
- recursive orchestration inflation
- uncontrolled parallelization

---

# Recommended Initial Flow

```txt
Human Request
  -> Carmilla Intake
    -> Executive Classification
      -> Staffing Resolution
        -> Chief Of Staff Activation
          -> Runtime Execution
            -> Artifact Production
              -> Verification
                -> Critique
                  -> Final Disposition
```

---

# Minimal Operational Team

## Required

- Carmilla
- Executive Layer
- Chief Of Staff
- Runtime Operator
- Verification Specialist

## Optional

- Critique Authority
- Auditor

---

# Required Artifacts

The Pilot Loop should minimally emit:

- intake packet
- staffing packet
- execution artifact
- verification artifact
- operational summary
- final disposition artifact

Operational reconstruction must remain possible.

---

# Runtime Requirements

The runtime must support:

- artifact persistence
- operational state emission
- verification visibility
- escalation visibility
- execution traceability

Silent execution is prohibited.

---

# Verification Requirements

Verification must:

- evaluate actual outputs
- preserve evidence lineage
- expose unresolved uncertainty
- reject fabricated confidence

Verification may initially remain lightweight.

It may not become ceremonial.

---

# Escalation Requirements

Escalation should occur when:

- staffing becomes ambiguous
- verification confidence drops
- runtime instability appears
- operational lineage becomes unclear
- consequence exposure increases

The Pilot Loop should prefer visible uncertainty over hidden instability.

---

# Restoration Expectations

The Pilot Loop should assume:

- failures will occur
- orchestration mistakes will occur
- continuity gaps will emerge

The objective is not perfection.

The objective is recoverable institutional coherence.

---

# Initial Suggested Missions

Recommended embodiment targets:

- repo analysis
- architecture review
- small implementation task
- documentation generation
- research synthesis
- orchestration planning

Avoid critical autonomous operations initially.

---

# Success Criteria

The Pilot Loop succeeds when:

- operational flow remains reconstructable
- staffing remains coherent
- verification remains honest
- escalation remains visible
- artifacts remain traceable
- final disposition remains justified

---

# Constitutional Rule

The Foundry becomes real only when operational doctrine survives contact with execution reality.
