# Foundry Staffing Engine

## Purpose

The Staffing Engine defines how The Foundry assembles operational structures from professions.

The Staffing Engine transforms objectives into:

- profession chains
- execution topology
- verification topology
- escalation topology
- coordination structures
- operational teams

---

# Core Principle

The Foundry staffs operational capability according to objective, consequence, dependency structure, and governance requirements.

---

# Staffing Inputs

The Staffing Engine should consider:

- operational objective
- consequence tier
- required artifacts
- dependency density
- orchestration complexity
- verification requirements
- escalation sensitivity
- runtime availability
- governance obligations

---

# Staffing Flow

## 1. Classification

Determine:

- strategic vs operational
- consequence tier
- governance sensitivity

Strategic situations should route upward before staffing decomposition.

---

## 2. Task Extraction

Extract:

- verb-plus-object task clusters
- required outputs
- operational dependencies

Tasks without clear objects remain unresolved.

---

## 3. Profession Resolution

Resolve each cluster into:

- profession
- authority envelope
- verification expectations
- runtime affinity

Possible resolution states:

- CLEAR
- AMBIGUOUS_LIGHT
- AMBIGUOUS_HEAVY
- NOVEL

Unresolved slots should not silently activate.

---

## 4. Dependency Topology

Determine:

- sequential dependencies
- parallelizable workstreams
- verification checkpoints
- escalation choke points
- integration points

Operational topology should remain reconstructable.

---

## 5. Coordination Analysis

Evaluate:

- dependency density
- orchestration complexity
- handoff fragility
- integration overhead

When coordination pressure exceeds threshold, add:

- coordinator
- executive secretary
- orchestration authority

---

## 6. Verification Topology

Determine:

- who executes
- who verifies
- who critiques
- who audits

Critical operations should avoid self-verification.

---

## 7. Activation Readiness

Determine:

- direct-fit installed skills
- reusable personas
- required materialization
- runtime compatibility
- governance readiness

Staffing recommendation is not activation.

---

# Staffing Outputs

The Staffing Engine should emit:

- ordered profession chain
- staffing topology
- dependency graph
- verification chain
- escalation topology
- activation readiness assessment
- orchestration handoff

---

# Minimum Viable Team Principle

Activate the smallest operational structure capable of completing the objective safely and coherently.

Avoid:

- orchestration inflation
- decorative roles
- unnecessary delegation
- overlapping ownership

Institutional elegance favors coherent minimality.

---

# Ownership Rules

Every staffed workstream should define:

- owner
- scope
- inputs
- outputs
- dependencies
- verification path
- completion signal

Ambiguous ownership destabilizes orchestration.

---

# Staffing Failure

Staffing failure occurs when:

- unresolved slots activate silently
- ownership overlaps dangerously
- verification chains collapse
- orchestration exceeds runtime capability
- dependency topology becomes incoherent
- governance obligations are ignored

Staffing failure should trigger escalation or replan.

---

# Relationship To The Citadel

The Foundry assembles.

The Citadel governs.

The Staffing Engine may not bypass:

- authority law
- verification doctrine
- consequence governance
- escalation doctrine
- trust disposition

---

# Constitutional Rule

Institutional strength emerges from coherent composition of operational capability.
